# 実装担当への指示書

あなたはこのリポジトリで、伺か風Live2Dデスクトップマスコットアプリ「ukaga」を実装する担当者です。
設計はすべて `plan/` ディレクトリに文書化済みです。**あなたの仕事は設計をコードにすることであり、設計を考え直すことではありません。**

## 最初にやること

1. `plan/` の全ファイルを番号順(01→07)に読む
2. `plan/07-roadmap.md` のフェーズ1から着手する

## 絶対に守るルール

- **計画に従う。** 技術スタック(Electron + TypeScript + electron-vite + pixi-live2d-display 等)、ディレクトリ構造、IPCチャンネル名、インターフェース定義は `plan/02〜05` に書かれたとおりに実装する。別のライブラリや構成に勝手に変更しない
- **フェーズ順に進める。** フェーズをまたいで先の機能を作り込まない。各フェーズ末尾の「確認」項目を満たしてから次へ進む
- **全部日本語。** コード内コメント・コミットメッセージ・README・エラーメッセージの解説はすべて日本語(`.claude/CLAUDE.md` 参照)
- **プロンプト文面をコードに埋め込まない。** キャラ人格・固定指示・ランダムトーク文面は `plan/04` の指定どおりテキストファイルに外部化する
- **1フェーズ = 1ブランチ or 一連のコミット**とし、フェーズ完了時にコミットする。コミットは動く状態で行う

## 判断に迷ったとき

- 計画に書いていない細部(変数名、細かいUIレイアウト等)は、計画の記述に最も近い素直な実装を選ぶ
- 計画どおりに実装できない事情(ライブラリのAPI変更、バージョン非互換など)が見つかったら:
  1. 勝手に設計変更せず、`plan/decisions.md` に「日付 / 問題 / 採った対応 / 影響範囲」を1エントリ追記する
  2. 影響が該当フェーズ内に収まる最小の回避策を選ぶ
- 計画同士が矛盾している場合は、より詳細な方(各論: 03〜06)を優先し、`plan/decisions.md` に記録する

## やってはいけないこと

- Ollama / VOICEVOX 等の外部ソフトのバイナリをリポジトリやアプリに同梱すること
- Live2D Cubism Core を npm 等の非公式ソースから取得すること(公式配布物を `resources/` に置く。入手手順はREADMEに書く)
- 設計にない設定項目・機能・依存パッケージの追加
- テストなしでのプロバイダ層(`llm/`・`voice/`)・文分割・感情タグ抽出の実装(ここは vitest ユニットテスト必須。HTTP はモックする)
- 動作確認せずに「完了」と報告すること

## 各フェーズの完了報告フォーマット

フェーズ完了時、次の形式で報告する:

```
## フェーズN 完了報告
- 実装した内容: (箇条書き)
- 動作確認: 07-roadmap.md の「確認」項目ごとに 実施方法と結果
- テスト: 追加したテストと `npm test` の結果
- 計画からの逸脱: なし / あり(decisions.md 参照)
- 既知の問題・次フェーズへの申し送り: (あれば)
```

## 環境に関する前提

- 開発マシンには Node.js(LTS)がある。パッケージマネージャは npm を使う
- 動作確認に Ollama と VOICEVOX が必要なフェーズ(2以降)では、起動していない場合のエラーハンドリング確認も「確認」項目に含める
- Live2Dサンプルモデルは Live2D 公式サイトから取得して `resources/models/` に配置する(取得手順を README に記載)
