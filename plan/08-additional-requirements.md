# 08. 追加要件・バックログ

フェーズ1〜の実装中に出た追加要望をここに記録する。  
本編計画(`01`〜`07`)のスコープ外だが、後続フェーズまたは計画改訂時に取り込む候補。

実装担当は、指示がない限り本ファイルの項目を勝手に実装しない。取り込み時は該当フェーズの計画に落とすか、本ファイルのステータスを更新する。

---

各項目は次の形式で追記する。

- **日付**: 要望が出た日
- **ステータス**: `未着手` / `計画取り込み済` / `完了` / `見送り`
- **内容**: 何をしたいか
- **現状**: いまの挙動
- **想定実装**: 方針のメモ（詳細は実装時に詰める）

### B-001: バルーンを明示的に閉じられるようにする

- **日付**: 2026-07-12
- **ステータス**: 完了
- **内容**: 一定時間待ちの自動フェード以外に、ユーザーがすぐバルーンを閉じられる操作を追加する
- **現状**: バルーン右上の「×」で即非表示。入力欄フォーカス中の Escape でも閉じる。外側クリック閉じはクリック透過のため不可
- **想定実装**（候補）:
  - バルーンに「閉じる」コントロールを置く（第一候補）
  - 入力欄フォーカス中の Escape で即非表示（補助）
  - 非表示中は既存どおりクリック透過。再表示はキャラクリック等（現行どおり）
  - 関連: [`03-mascot-live2d.md`](03-mascot-live2d.md) バルーンUI、「一定時間操作がなければフェードアウト」
- **制約（2026-07-12 レビューで確認）**:
  - 「バルーン外（キャラ外のウィンドウ領域）クリックで閉じる」は**不可**。キャラ外の領域は `setIgnoreMouseEvents(true)` でクリックが下のアプリへ素通しされるため、ウィンドウ側はそのクリックを受け取れない（`plan/03` クリック透過設計）
  - Escape はマスコットウィンドウにキーボードフォーカスがあるとき（実質、入力欄フォーカス中）しか届かない。グローバルショートカット化は他アプリと衝突するため採らない

### B-002: 管理画面にタブごとの明示的な保存ボタンを置く

- **日付**: 2026-07-12
- **ステータス**: 完了
- **内容**: 各設定タブ（LLM / 音声 / キャラクター / 挙動）で、変更内容をはっきり保存できるボタンがほしい。いまの即時反映だけだと「保存できたか」が分かりにくい
- **現状**: 即時反映は維持し、設定・人格・モデル変更の成功時に右下へ「保存済み ✓」トーストを表示
- **想定実装**（候補）:
  - 各タブに「このタブを保存」ボタンを置く（または画面下部の共通保存）
  - 編集中はローカル state に留め、保存押下でメインへ送る
  - 未保存時は離脱警告や「未保存」表示があるとよい
  - 即時反映を残すか、保存ボタン方式に切り替えるかは取り込み時に決める（現行計画は即時反映）
  - **軽量代替案**: 要望の本質が「保存できたか分かりにくい」であれば、即時反映のまま保存成功時に「保存済み ✓」トースト/インジケータを出すだけで足りる可能性がある。まずこちらを検討する ← **採用**
  - 関連: [`06-admin-ui.md`](06-admin-ui.md) 設定反映の仕組み
- **取り込み時の論点（2026-07-12 レビューで確認）**:
  - persona.md は外部エディタ編集を `fs.watch` で監視し `persona:changed` で管理画面へ反映する設計（`plan/04`）。保存ボタン方式にすると「未保存のローカル state」と「外部でのファイル変更」が衝突するため、解決ルール（例: 未編集なら外部変更を自動反映、編集中なら「外部で変更されました: 破棄して再読込 / 自分の編集を維持」の選択を出す）を決める必要がある
  - 軽量案採用により上記衝突は回避

### B-007: バルーンの可動範囲が狭い（上下とも）

- **日付**: 2026-07-12
- **ステータス**: 完了
- **内容**: バルーンをもっと広い範囲（特に上下）に動かせるようにする
- **現状**: 候補1を採用。挙動タブの「ウィンドウ」節でウィンドウ幅・高さ（px）を変更可能（下限240×320、作業領域上限、足元=下端中央を保ってリサイズ）。ウィンドウを広げるとバルーン可動域がそのまま広がる。キャラサイズはウィンドウから分離済み（B-008参照）のため、広げてもキャラは大きくならない
- **根本原因**: ウィンドウサイズが固定で、バルーンがその中に閉じ込められている（B-008と共通）
- **想定実装**（候補）:
  1. **ウィンドウサイズを設定可能にする（第一候補）**: `window.width/height` は設定スキーマに既にあるので、管理画面（挙動タブ等）から変更できるようにし、`resizable` は false のまま `setSize` で反映。ウィンドウを縦横に広げればバルーン可動域がそのまま広がる。透明領域が増えてもキャラ外はクリック透過済みなので実害は小さい
  2. バルーンを独立した `BrowserWindow` に分離し、画面全体を可動域にする。ただし `plan/03` の「同一ウィンドウ内DOM」決定を覆すことになり、位置同期・OS差異対応のコストが大きい。1で不足する場合の将来案
- **注意**: バルーンはDOMである以上「ウィンドウ外へのはみ出し」は原理的に不可。クランプ緩和だけでは解決しない
- 関連: [`03-mascot-live2d.md`](03-mascot-live2d.md) バルーンUI、B-006、B-008（同根）

### B-008: キャラの表示サイズを実質変更できない

- **日付**: 2026-07-12
- **ステータス**: 完了
- **内容**: デスクトップ上でのキャラの大きさを自由に変えられるようにする
- **現状**: 候補1+2を採用。キャラのレイアウト基準を「実ウィンドウ」から「固定の基準ボックス（420×880）」に変更し、画面上のキャラサイズはスケール設定だけで決まるようにした（`stage.ts`）。スケール拡大でキャラがウィンドウに収まらない場合は、レンダラが `mascot:ensureWindowSize` でウィンドウを自動拡大（拡大のみ・足元固定・作業領域内にクランプ・設定に永続化）。ウィンドウサイズとスケールは独立した設定として管理画面から変更可能
- **根本原因**: ウィンドウサイズが固定（B-007と共通）
- **想定実装**（候補）:
  1. **スケール変更にウィンドウサイズを連動させる（第一候補）**: スケール変更時に基準サイズ×スケールで `setSize` し、画面上のキャラサイズが実際に変わるようにする。バルーン用上部ゾーン（`TOP_BALLOON_ZONE_*`）分も加算。位置は下端基準を維持（足元がずれないように）
  2. B-007案1のウィンドウサイズ設定と統合し、「ウィンドウサイズ」と「スケール」を独立した設定として両方管理画面に出す
  3. （追加候補）キャラ上での Ctrl/Alt+ホイールでその場リサイズ（伺か的な操作感。1が入っていれば小さい追加）
- **注意**: ウィンドウ拡大時は作業領域（`workAreaSize`）を超えないようクランプし、画面外に出た場合は位置補正する
- 関連: [`03-mascot-live2d.md`](03-mascot-live2d.md)、[`06-admin-ui.md`](06-admin-ui.md) 表示スケール、B-007（同根）

### B-009: バルーン位置をキャラサイズ（ウィンドウリサイズ）と連動せず維持する

- **日付**: 2026-07-12
- **ステータス**: 対応不要
- **内容**: キャラの表示スケール変更や、管理画面からのウィンドウサイズ変更によってウィンドウがリサイズ（下端中央基準）された際、バルーンのデスクトップ上の表示位置（絶対位置）がずれないように維持する
- **現状**: ウィンドウサイズが変更されると、ウィンドウの左上（原点）が移動するため、ウィンドウ左上からの相対座標で配置されているバルーンはデスクトップ上で位置がずれてしまう（キャラのサイズ変更に連動して動いてしまう）
- **想定実装**（候補）:
  - レンダラーの `onSettingsChanged` でウィンドウサイズ（`settings.window` の幅・高さ）の変化を検知する
  - ウィンドウサイズの変化量（幅の差分の半分、高さの差分）をバルーンの相対座標に加算して補正し、適用および設定保存を行う
  - 自身で位置を保存した際のフィードバック（設定変更通知）で座標が古いもので上書きされないよう、ガード処理（`lastSavedBalloonPos` による判定など）を入れる
  - 関連: B-007、B-008

### B-010: 設定の自動保存廃止（保存ボタン適用）と入力操作性の改善

- **日付**: 2026-07-12
- **ステータス**: 完了
- **内容**: 設定画面での「入力ごとの自動保存（即時反映）」を廃止し、各タブ（または設定画面全体）に「保存」ボタンを設置して、明示的にボタンを押下したタイミングで全設定（数値・テキスト・選択肢等すべて）を適用・保存する仕様に変更する。これに伴い、数値入力フィールドでタイピングする際のスムーズな入力も可能にする。
- **現状**:
  - `onChange` イベントのたびに IPC を介して設定保存・即時反映が走るため、入力カクつきやフォーカス喪失が発生しやすい。
  - 特に数値入力（履歴送信数、ランダムトーク間隔、ウィンドウ幅・高さ、話者ID）では、入力途中で値が強制的に最小値にリセットされるため操作性が極めて悪い。
- **想定実装**（候補）:
  - 自動保存（`onChange` での `patchSettings` 呼び出し）を完全に廃止する。
  - 各タブに「設定を保存」ボタンを配置する（または管理画面共通の保存フッターを置く）。
  - 入力中はすべての設定項目（テキスト、数値、チェックボックス等）の変更を React のローカル state に留め、保存ボタンが押されたタイミングで一括バリデーション（数値のクランプ等）を行い、メインプロセスに設定を反映・保存する。
  - 未保存の変更がある状態でタブを切り替えたり画面を閉じようとした場合の警告や、変更検知（`dirty` 判定）を入れるとより親切。
  - 関連: B-002（当初の軽量代替案を廃止し本格対応）、[`06-admin-ui.md`](06-admin-ui.md)
